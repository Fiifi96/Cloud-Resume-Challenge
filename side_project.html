Website Architecture

This document describes a sophisticated DevOps project, primarily developed with HTML and CSS, that operates via Docker containers and employs continuous integration/continuous deployment (CI/CD) practices, utilizing tools such as NGINX, Jenkins, Docker, DockerHub, and GitHub. The project is structured to incorporate an automated pipeline, whereby the code's transition from GitHub repository to the final production server is seamlessly executed through a series of stages.

Website and Server Architecture

The project revolves around a web interface, composed of numerous HTML files. The entry point to this website is landing.html, a dedicated home page. In terms of infrastructure, the website is hosted by an NGINX server, packaged within a Docker container. This setup ensures a uniform, reproducible environment and facilitates straightforward portability and scaling.

Code Repository

All project-related code is stored in a private GitHub repository (Cloud-Resume-Challenge). This version control system ensures a robust, organized, and collaborative approach to development, as well as secure storage of all project code. It also forms the basis for the subsequent CI/CD stages, as code updates in this repository automatically initiate the pipeline.

Continuous Integration / Continuous Deployment

This project incorporates a robust CI/CD pipeline, powered by Jenkins, to automate the integration, testing, and deployment stages. The pipeline is initiated whenever code is committed and pushed to the remote GitHub repository.

The pipeline consists of three main jobs:

  1. Code Integration: The initial job begins by cloning the repository onto the Jenkins server. Following this, the Docker image is built. This includes downloading dependencies, running build scripts, and generating an executable Docker container that encapsulates the NGINX server with the website code.
  2. Testing and DockerHub Upload: The second job takes the built Docker image and runs it in a contained environment, executing a series of tests to ensure that the system behaves as expected. Once the image passes the testing phase, it is pushed to DockerHub, a publicly available container registry service. DockerHub serves as a centralized repository for Docker images, ensuring they are accessible for deployment.
  3. Deployment: The final job in the pipeline is deployment. This involves pulling the Docker image from DockerHub and running it on the production server (Azure, AWS, GCP). The server then launches the Docker container, effectively deploying the updated website code in a production environment.

Through these automated steps, the project maintains a consistent, reliable, and efficient deployment process, reducing manual intervention and potential for error.
